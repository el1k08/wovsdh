# Database Schema Documentation

## Overview

This document describes the PostgreSQL schema (hosted on Supabase) for the nail service booking system. The system manages two studios, time slots, client bookings, staff authorization, and email audit logs.

All timestamps are stored as `TIMESTAMPTZ` (UTC) and converted to `Asia/Jerusalem` for display. Primary keys use UUIDs generated by `gen_random_uuid()`.

### Race Condition Prevention

**Problem:** Two concurrent clients could simultaneously submit bookings for the same slot. Between the moment client A reads the slot as `available` and the moment their booking is written, client B can do the same. Without a database-level constraint both inserts would succeed, producing a double-booking.

**Solution â€” Partial Unique Index:**

```sql
CREATE UNIQUE INDEX uq_bookings_slot_active
    ON bookings (slot_id)
    WHERE status != 'CANCELLED';
```

This index guarantees that at most one row with a given `slot_id` can exist in a non-cancelled state. If two concurrent transactions both attempt to `INSERT` a booking for the same `slot_id`, PostgreSQL's MVCC serialization ensures only one succeeds; the second receives a `unique_violation` error (code `23505`), which the API layer catches and converts into a `409 Conflict` response.

**Why not a DB trigger or application-level lock?**
- Application-level locks (e.g. Redis SETNX) introduce an external dependency and can be bypassed if requests hit different API instances without coordination.
- A DB trigger that checks for existing active bookings is subject to a TOCTOU race under `READ COMMITTED` isolation.
- The partial unique index is enforced atomically by the PostgreSQL storage engine regardless of isolation level, transaction order, or which API instance handles the request. It is the only truly race-safe solution within PostgreSQL.

**Slot status transition:**
When a booking is created (`PENDING`), the slot's `status` is atomically updated to `booked` in the same transaction. This prevents the slot from appearing in the public available-slots query immediately.

## Row Level Security (RLS) Policies

All tables have RLS enabled. The application backend uses the `service_role` key (bypasses RLS) for write operations. The public-facing Next.js API uses the `anon` key only for reading available slots.

### `studios`

| Policy Name          | Role    | Operation | Using Expression | Description                          |
|----------------------|---------|-----------|------------------|--------------------------------------|
| `studios_read_all`   | `anon`  | SELECT    | `TRUE`           | Anyone can read studio reference data |

### `slots`

| Policy Name                  | Role    | Operation | Using Expression              | Description                                    |
|------------------------------|---------|-----------|-------------------------------|------------------------------------------------|
| `slots_read_available`       | `anon`  | SELECT    | `status = 'available'`        | Anonymous users see only bookable slots        |
| `slots_service_role_all`     | `service_role` | ALL | `TRUE`                  | Service role has full access                   |

Rationale: clients must not be able to read `blocked` or `booked` slots' details, and must never be able to write directly to this table.

### `bookings`

| Policy Name                  | Role           | Operation | Using Expression | Description                              |
|------------------------------|----------------|-----------|------------------|------------------------------------------|
| `bookings_service_role_all`  | `service_role` | ALL       | `TRUE`           | Only the backend service can read/write bookings |

Rationale: bookings contain personal client data (name, phone, email). No anonymous or authenticated-user policies are defined; all access goes through Next.js API routes that use the service role key server-side.

### `allowed_users`

| Policy Name                      | Role           | Operation | Using Expression | Description                               |
|----------------------------------|----------------|-----------|------------------|-------------------------------------------|
| `allowed_users_service_role_all` | `service_role` | ALL       | `TRUE`           | Only backend service can manage the whitelist |

### `email_logs`

| Policy Name                   | Role           | Operation | Using Expression | Description                      |
|-------------------------------|----------------|-----------|------------------|----------------------------------|
| `email_logs_service_role_all` | `service_role` | ALL       | `TRUE`           | Only backend service writes logs |

## Triggers

### `set_updated_at`

A generic `BEFORE UPDATE` trigger applied to the `bookings` table. On every UPDATE it sets `updated_at = NOW()`, removing the need for the application layer to manage this field.

```sql
CREATE OR REPLACE FUNCTION trigger_set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```
