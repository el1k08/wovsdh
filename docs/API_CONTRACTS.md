# API Contracts

All endpoints are Next.js API Routes served from `/api/`. The backend uses the Supabase `service_role` key for all database operations. No client-facing endpoint exposes raw Supabase credentials.

**Base URL:** `https://<domain>/api`  
**Content-Type:** `application/json`  
**Timezone:** All `datetime` values are ISO 8601 with UTC offset. The API accepts and returns UTC; the frontend converts to `Asia/Jerusalem` for display.

---

## Error Response Envelope

All error responses follow this shape:

```json
{
  "error": {
    "code": "SLOT_UNAVAILABLE",
    "message": "The selected slot is no longer available."
  }
}
```

**Standard HTTP error codes used:**

| Code | Meaning                                                         |
|------|-----------------------------------------------------------------|
| 400  | Bad Request — missing or invalid input fields                   |
| 404  | Not Found — resource does not exist                             |
| 409  | Conflict — race condition / double-booking attempt              |
| 422  | Unprocessable Entity — business rule violation                  |
| 500  | Internal Server Error — unexpected failure                      |

## Public Endpoints

---

### `GET /api/slots`

Returns available time slots for a studio on a given date. Only slots with `status = 'available'` are returned.

**Query Parameters:**

| Parameter  | Type   | Required | Description                         |
|------------|--------|----------|-------------------------------------|
| `studio_id`| string | Yes      | `rishon` or `ashdod`                |
| `date`     | string | Yes      | ISO date `YYYY-MM-DD` in local time |

**Success Response `200 OK`:**

```json
{
  "slots": [
    {
      "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
      "studio_id": "rishon",
      "start_at": "2024-03-15T09:00:00+02:00",
      "end_at": "2024-03-15T10:00:00+02:00",
      "status": "available"
    },
    {
      "id": "b2c3d4e5-f6a7-8901-bcde-f12345678901",
      "studio_id": "rishon",
      "start_at": "2024-03-15T10:00:00+02:00",
      "end_at": "2024-03-15T11:00:00+02:00",
      "status": "available"
    }
  ]
}
```

**Error Responses:**

```json
// 400 — missing or invalid studio_id
{
  "error": {
    "code": "INVALID_PARAMS",
    "message": "studio_id must be one of: rishon, ashdod"
  }
}

// 400 — invalid date format
{
  "error": {
    "code": "INVALID_PARAMS",
    "message": "date must be in YYYY-MM-DD format"
  }
}
```

**Implementation Notes:**
- The query filters: `studio_id = $1 AND start_at >= $2 AND start_at < $3 AND status = 'available'`
  where `$2` is the start of the requested date in `Asia/Jerusalem` converted to UTC and `$3` is the start of the next day.
- Uses `idx_slots_studio_start_status` index.

### `POST /api/bookings`

Creates a new booking. The slot status is atomically updated to `booked` in the same transaction. A Telegram notification is dispatched to all active staff members.

**Request Body:**

```json
{
  "slot_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
  "studio_id": "rishon",
  "client_first_name": "Мария",
  "client_last_name": "Иванова",
  "client_phone": "+972501234567",
  "client_email": "maria@example.com"
}
```

**Success Response `201 Created`:**

```json
{
  "booking": {
    "id": "c3d4e5f6-a7b8-9012-cdef-123456789012",
    "slot_id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
    "studio_id": "rishon",
    "status": "PENDING",
    "start_at": "2024-03-15T09:00:00+02:00",
    "end_at": "2024-03-15T10:00:00+02:00",
    "created_at": "2024-03-14T17:23:11Z"
  }
}
```

**Error Responses:**

```json
// 400 — validation failure
{
  "error": {
    "code": "VALIDATION_ERROR",
    "message": "client_email is required and must be a valid email address"
  }
}

// 404 — slot does not exist
{
  "error": {
    "code": "SLOT_NOT_FOUND",
    "message": "The requested slot does not exist."
  }
}

// 409 — race condition: slot was booked between client reading and submitting
{
  "error": {
    "code": "SLOT_UNAVAILABLE",
    "message": "The selected slot is no longer available. Please choose another time."
  }
}
```

**Implementation Notes:**
- The DB transaction: `BEGIN` → update `slots.status = 'booked'` → `INSERT INTO bookings` → `COMMIT`.
- The partial unique index `uq_bookings_slot_active` catches concurrent inserts and raises `23505`; the handler translates this to `409`.
- Telegram notification is sent after the transaction commits (fire-and-forget; failure does not roll back the booking).
- The `cancellation_token` is generated by the DB (`DEFAULT gen_random_uuid()`); it is never returned to the client in this response — it is embedded in the email sent on confirmation.

### `GET /api/bookings/[token]`

Fetches booking details by cancellation token. Used by the cancellation page to display booking information before the client confirms cancellation.

**Path Parameter:**

| Parameter | Type   | Description                       |
|-----------|--------|-----------------------------------|
| `token`   | UUID   | `cancellation_token` from the email link |

**Success Response `200 OK`:**

```json
{
  "booking": {
    "id": "c3d4e5f6-a7b8-9012-cdef-123456789012",
    "studio_id": "rishon",
    "studio_name": "Студия маникюра Ришон-ле-Цион",
    "client_first_name": "Мария",
    "status": "CONFIRMED",
    "start_at": "2024-03-15T09:00:00+02:00",
    "end_at": "2024-03-15T10:00:00+02:00"
  }
}
```

**Error Responses:**

```json
// 404 — token not found or booking already cancelled
{
  "error": {
    "code": "BOOKING_NOT_FOUND",
    "message": "No active booking found for this link."
  }
}
```

**Implementation Notes:**
- Returns 404 for both non-existent tokens and already-cancelled bookings (avoids information leakage about past booking existence).
- Does not return `client_email`, `client_phone`, or `client_last_name` (minimal exposure).

---

### `POST /api/bookings/cancel`

Cancels a booking identified by its cancellation token. Does not require authentication.

**Request Body:**

```json
{
  "token": "d4e5f6a7-b8c9-0123-defa-234567890123"
}
```

**Success Response `200 OK`:**

```json
{
  "message": "Your booking has been successfully cancelled.",
  "booking": {
    "id": "c3d4e5f6-a7b8-9012-cdef-123456789012",
    "status": "CANCELLED",
    "cancelled_at": "2024-03-14T18:45:00Z"
  }
}
```

**Error Responses:**

```json
// 404 — token not found
{
  "error": {
    "code": "BOOKING_NOT_FOUND",
    "message": "No active booking found for this cancellation link."
  }
}

// 422 — booking already cancelled
{
  "error": {
    "code": "ALREADY_CANCELLED",
    "message": "This booking has already been cancelled."
  }
}
```

**Implementation Notes:**
- Transaction: `UPDATE bookings SET status = 'CANCELLED', cancelled_at = NOW() WHERE cancellation_token = $1 AND status != 'CANCELLED'`.
- After commit: update `slots.status = 'available'`, delete the Google Calendar event (if `google_calendar_event_id` is set), send a cancellation email, notify staff via Telegram.
- All post-commit side effects are fire-and-forget; failures are logged to `email_logs` and application logs but do not affect the 200 response.

## Telegram Webhook

---

### `POST /api/telegram/webhook`

Receives Telegram Bot API update events. All requests are validated against the `X-Telegram-Bot-Api-Secret-Token` header.

**Headers:**

| Header                              | Value                             |
|-------------------------------------|-----------------------------------|
| `X-Telegram-Bot-Api-Secret-Token`   | Pre-shared secret set in BotFather |

**Telegram Update Payload (Telegram sends this):**

```json
{
  "update_id": 123456789,
  "callback_query": {
    "id": "callback_query_id",
    "from": {
      "id": 987654321,
      "username": "staffmember"
    },
    "message": {
      "message_id": 42,
      "chat": { "id": 987654321 }
    },
    "data": "confirm:c3d4e5f6-a7b8-9012-cdef-123456789012"
  }
}
```

**Callback Data Format:**

| Action    | `data` value                         |
|-----------|--------------------------------------|
| Confirm   | `confirm:<booking_id>`               |
| Cancel    | `cancel:<booking_id>`                |

**Success Response `200 OK`:**

```json
{ "ok": true }
```

Telegram requires a 200 response within 5 seconds or it will retry the webhook.

**Error Response (still returns 200 to prevent Telegram retries):**

```json
{ "ok": false, "error": "UNAUTHORIZED" }
```

**Authorization Flow:**
1. Extract `from.id` (the Telegram `chat_id`) from the update.
2. Query `allowed_users` where `telegram_chat_id = $1 AND is_active = TRUE`.
3. If not found: answer the callback with an "Unauthorized" message, return `{ "ok": false }`.
4. If found: process the action.

**Confirm Action:**
1. Set `bookings.status = 'CONFIRMED'`, `confirmed_at = NOW()`.
2. Create Google Calendar event → store `google_calendar_event_id` in the booking.
3. Send confirmation email to the client (appends cancellation link with `cancellation_token`).
4. Log to `email_logs`.
5. Edit the Telegram message to show "Confirmed by <name>" (uses `telegram_message_id`).

**Cancel Action (staff-initiated):**
1. Set `bookings.status = 'CANCELLED'`, `cancelled_at = NOW()`.
2. Update `slots.status = 'available'`.
3. Delete Google Calendar event if present.
4. Send cancellation email to client.
5. Edit the Telegram message to show "Cancelled by <name>".

## Admin Endpoints

These endpoints require the `Authorization: Bearer <ADMIN_SECRET>` header. The `ADMIN_SECRET` is an environment variable. There is no user-facing login; admin access is token-gated.

---

### `POST /api/admin/slots/generate`

Generates a batch of slots for a studio. Idempotent: existing slots in the range are skipped (not duplicated).

**Request Body:**

```json
{
  "studio_id": "rishon",
  "date": "2024-03-15",
  "slot_duration_minutes": 60,
  "start_time": "09:00",
  "end_time": "18:00"
}
```

**Success Response `201 Created`:**

```json
{
  "created": 9,
  "skipped": 0,
  "slots": [
    {
      "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890",
      "start_at": "2024-03-15T07:00:00Z",
      "end_at": "2024-03-15T08:00:00Z",
      "status": "available"
    }
  ]
}
```

**Error Responses:**

```json
// 400 — invalid time range
{
  "error": {
    "code": "INVALID_PARAMS",
    "message": "end_time must be after start_time"
  }
}

// 401 — missing or invalid admin secret
{
  "error": {
    "code": "UNAUTHORIZED",
    "message": "Valid admin credentials are required."
  }
}
```

**Implementation Notes:**
- `start_time` and `end_time` are local times in `Asia/Jerusalem`.
- The handler converts them to UTC before inserting.
- Uses `INSERT ... ON CONFLICT DO NOTHING` against a unique constraint on `(studio_id, start_at)` to ensure idempotency.

### `DELETE /api/admin/slots/[id]`

Deletes a slot. Only allowed if the slot has no active (non-cancelled) booking. Slots with active bookings must be cancelled through the booking cancellation flow first.

**Path Parameter:**

| Parameter | Type | Description  |
|-----------|------|--------------|
| `id`      | UUID | Slot UUID    |

**Success Response `200 OK`:**

```json
{
  "message": "Slot deleted successfully.",
  "id": "a1b2c3d4-e5f6-7890-abcd-ef1234567890"
}
```

**Error Responses:**

```json
// 404 — slot not found
{
  "error": {
    "code": "SLOT_NOT_FOUND",
    "message": "Slot not found."
  }
}

// 422 — slot has an active booking
{
  "error": {
    "code": "SLOT_HAS_ACTIVE_BOOKING",
    "message": "Cannot delete a slot with an active booking. Cancel the booking first."
  }
}
```

## Webhook Security

| Endpoint              | Auth Mechanism                                      |
|-----------------------|-----------------------------------------------------|
| `/api/telegram/webhook` | `X-Telegram-Bot-Api-Secret-Token` header comparison |
| `/api/admin/*`        | `Authorization: Bearer <ADMIN_SECRET>` header       |
| All public endpoints  | None (rate-limited at infrastructure level)         |

Rate limiting for public endpoints should be applied at the infrastructure layer (e.g. Vercel Edge Config, Cloudflare). The API itself does not implement rate limiting to avoid complexity at the application layer.
